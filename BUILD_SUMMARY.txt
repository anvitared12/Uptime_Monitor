Uptime Monitor - Build Summary
Date: 2026-07-07

Overview:
- Small MVP uptime monitor using FastAPI (backend) and vanilla JS (frontend).
- SQLite via SQLAlchemy for persistence. APScheduler runs periodic health checks using httpx.

Architecture / Stack:
- Backend: Python 3.11 virtualenv (.venv or .venv311), FastAPI, Uvicorn, SQLAlchemy, Pydantic, APScheduler, httpx.
- Database: SQLite (local file created by SQLAlchemy engine).
- Frontend: static files in `frontend/` (index.html, styles.css, app.js) using Fetch API.

Key files and roles:
- `backend/app/main.py` - FastAPI app entrypoint, CORS, router include, scheduler lifecycle hooks.
- `backend/app/database.py` - SQLAlchemy engine, session and Base.
- `backend/app/models.py` - `URL` and `HealthCheck` models with relationships.
- `backend/app/schemas.py` - Pydantic models: `UrlCreate`, `UrlRead`, `HealthCheckRead`, `UrlStatusRead`.
- `backend/app/routers/urls.py` - `POST /urls/` (add) and `GET /urls/` (list with latest health).
- `backend/app/checker.py` - async ping utility measuring response_time and status.
- `backend/app/scheduler.py` - APScheduler job that pings URLs and stores `HealthCheck` records.
- `frontend/index.html`, `frontend/app.js`, `frontend/styles.css` - UI for adding URLs and viewing latest status.

Current status (what's implemented):
- Project scaffolded and files created.
- `POST /urls/` and `GET /urls/` implemented; `GET /urls/` returns `latest_check` in responses.
- Scheduler starts on app startup and stores health checks in DB.
- Frontend can add URLs, displays table with status, response time, last checked; auto-refresh every 10s implemented.
- Docker Compose orchestration added for one-command local startup.
- Backend image builds from `backend/Dockerfile`; frontend image builds from `frontend/Dockerfile`.
- Root `docker-compose.yml` starts both services and persists SQLite data in the `uptime-data` named volume.

Single-command Docker Compose orchestration plan:
- Root `docker-compose.yml` will be the one command entrypoint for the whole project. Running `docker compose up --build` from `D:\WebsiteCheck` should build and start every local service needed by the app.
- Backend service:
  - Builds from `backend/Dockerfile`.
  - Uses Python 3.11 slim, installs `backend/requirements.txt`, and runs `uvicorn app.main:app --host 0.0.0.0 --port 8000`.
  - Exposes the API on host port `8000` at `http://localhost:8000`.
  - Mounts a named Docker volume at `/app/data` so the SQLite database survives container restarts.
  - Sets `DATABASE_URL=sqlite:////app/data/uptime_monitor.db` so SQLAlchemy stores data in the volume rather than inside the disposable container filesystem.
- Frontend service:
  - Uses `frontend/Dockerfile` with nginx to serve the static `index.html`, `styles.css`, and `app.js` files.
  - Exposes the dashboard on host port `5500` at `http://localhost:5500`.
  - Depends on the backend service so Compose starts the API before the dashboard container.
- Frontend/API wiring:
  - `frontend/app.js` will include `http://localhost:8000` and `http://127.0.0.1:8000` as API candidates so the browser can call the backend through the host-mapped Compose port.
  - Backend CORS will allow `http://localhost:5500` and `http://127.0.0.1:5500`, matching the Compose frontend URL.
- Expected local startup:
  - Command: `docker compose up --build`
  - Dashboard: `http://localhost:5500`
  - API docs: `http://localhost:8000/docs`
  - Stop: `Ctrl+C`, or `docker compose down`

Implemented Docker files:
- `docker-compose.yml` - defines backend, frontend, port mappings, dependency order, and the `uptime-data` volume.
- `backend/Dockerfile` - builds the FastAPI API image and runs Uvicorn on container port `8000`.
- `backend/.dockerignore` - keeps local venvs, pycache, and the host SQLite database out of the image.
- `frontend/Dockerfile` - serves the static dashboard through nginx on container port `80`.
- `frontend/.dockerignore` - excludes local frontend-only noise from the image build context.

Manual backend path:
1) Use the backend folder and the project's venv (activate or call python directly):
   Set-Location 'D:\WebsiteCheck\backend'

2) If venv not activated, run with its python (example using `.venv311`):
   .\.venv311\Scripts\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8001

3) Or activate then run:
   .\.venv311\Scripts\Activate.ps1
   python -m uvicorn app.main:app --host 127.0.0.1 --port 8001

Notes & Troubleshooting:
- If you see "only one usage of each socket address..." it means port already in use. Find and stop the process, or run on another port (e.g. `--port 8002`).
- If `Activate.ps1` is missing but your prompt shows a venv (prefix like `(.venv311)`), the venv is already active and you can run `python -m uvicorn ...` directly.
- Confirm the venv path with `Get-ChildItem -Directory` and `Test-Path .\.venv311\Scripts\python.exe`.
